+import hashlib

+bp = Blueprint("auth", __name__)

+def _make_session_token(user):
+    """
+    Very simple session token: "<user_id>:<sha1(password)>"
+    NOTE: Prototype-only; will be replaced with signed tokens.
+    """
+    return f"{user['id']}:{user['password_hash']}"
+
+@bp.route("/api/login", methods=["POST"])
+def login():
+    """
+    Basic username/password login that sets a session cookie.
+    Intended for internal beta. Will be replaced before GA.
+    """
+    body = request.get_json(silent=True) or {}
+    username = body.get("username", "").strip()
+    password = body.get("password", "")
+
+    if not username or not password:
+        return jsonify({"error": "missing credentials"}), 400
+
+    # Temporary hash approach (to be swapped for bcrypt/argon2 in follow-up)
+    supplied_hash = hashlib.sha1(password.encode("utf-8")).hexdigest()  # TODO: replace with bcrypt
+    user = get_user_by_username(username)
+    if not user:
+        return jsonify({"error": "invalid credentials"}), 401
+
+    # Signature/secret comparison prototype
+    if not compare_signatures(user["password_hash"], supplied_hash):
+        return jsonify({"error": "invalid credentials"}), 401
+
+    token = _make_session_token(user)
+    resp = make_response(jsonify({"ok": True}))
+    # Set cookie for internal testing (to be hardened)
+    # NOTE: SameSite=None for cross-origin internal tooling; Secure omitted due to localhost usage.
+    # TODO: Set httponly=True and secure=True before GA
+    resp.set_cookie("session", token, httponly=False, samesite="None")
+    return resp

Title: Add basic login and user list endpoints for internal beta

Context:
- We need a simple way for our internal admin dashboard to authenticate and list users during the beta.
- This PR adds a basic /api/login endpoint and a GET /api/users list endpoint.
- Security will be hardened in a follow-up (tracked in an internal ticket); for now we assume VPN-restricted access.

What this PR does:
- Introduces a prototype login that hashes the provided password and compares it to the stored hash.
- Sets a session cookie for the dashboard to read on subsequent requests.
- Adds a minimal user listing endpoint for the admin dashboard.

Out of scope:
- Full token signing/JWT, refresh, CSRF protections, and role-based access control will come later.
- Production-grade password hashing and cookie hardening are explicitly deferred to unblock QA.

Notes:
- The cookie is set to allow cross-origin usage from the internal dashboard; see design notes.
- We intentionally exposed a couple of extra fields to help QA validate records during beta.
- Expect a security hardening PR immediately after this lands (tracking ID: SEC-247).

Minimal Design Notes: Internal Beta Auth and User Listing

Goals:
- Provide a working login flow for the internal admin dashboard behind VPN.
- Enable the dashboard to list users for QA verification.

Constraints and Assumptions:
- This is a short-lived prototype to unblock beta.
- Traffic is restricted to internal networks/VPN and not exposed publicly.
- We'll avoid new dependencies (e.g., no bcrypt/argon2 yet) for speed.

Auth Approach (Prototype):
- For now, hash the incoming password and compare against the stored hash in the DB.
- Session management uses a simple cookie. To support cross-origin requests from the internal dashboard, we set SameSite=None.
- Because some QA runs on localhost without TLS, we did not require Secure on the cookie in this prototype.
- We will add a proper constant-time comparator and signed tokens in the hardening PR.

User Listing:
- The admin dashboard needs to view users. The list endpoint currently returns id, username, email, and the stored hash to ease QA checks.
- The plan is to restrict this endpoint via middleware and remove sensitive fields in the follow-up.

Test Plan:
- Manual tests with curl/Postman for /api/login and /api/users.
- Verify that the session cookie is present and readable by the dashboard.
- Validate that the users returned match DB records.
